/loop loop-hardener

Start from {approved_loop}. Improve one approved /loop by moving its task-value/resource-cost frontier. A valid win either increases verified task value without unacceptable added cost, or lowers resource cost without losing task value, proof, reliability, or required complexity. Cost is not the universal objective. Addition, removal, restructuring, verification, routing, stabilization, and deliberate preservation are all valid operations when they address the current bottleneck. Do not optimize one convenient proxy until the workflow collapses into less work, less evidence, less depth, or less product.

If the caller supplies {campaign_objective}, treat it as the campaign direction and ledger, not permission to stop. Without one, default to MULTI_CYCLE. The operator is the stop condition. Keep challenger verdicts separate from campaign state: NO PROMOTION, INCONCLUSIVE, NEEDS_MEASUREMENT, ECONOMICALLY_EXHAUSTED, or a rejected challenger resolves that cycle or lane only and never completes the campaign.

Confirm that the target is APPROVED or explicitly APPROVED-FOR-HARDENING. Approval for hardening is not canonical approval. If neither exists, return NEEDS_APPROVAL without executing the target. Create {dir}/hardening/{loop_name}/, copy the target to baseline.md, record its hash, make it read-only when possible, and never edit it. Every change belongs in a challenger copy.

Before running the target, classify side effects as READ_ONLY, ISOLATED_WRITE, or PRODUCTION_WRITE. Any loop that can deploy, send, publish, mutate external data, touch customer-like data, spend money, delete data, change schedulers, modify production files, or otherwise affect the outside world is PRODUCTION_WRITE unless definitively isolated. When uncertain, classify upward. Do not benchmark PRODUCTION_WRITE live without exact approval. Use controlled clones, disposable tenants, fixtures, intercepted outbound actions, draft-only external outputs, or explicit one-time authorization for the exact external action. A substitute must be executed and behavior-preserving; it proves only the exercised path, and untested real behavior remains NEEDS_REAL_RUN or NEEDS_MEASUREMENT. HUMAN-GATED promotion authority does not authorize live execution.

Write hardening_manifest.md before challenger work. Record approval and baseline hash, campaign direction, operator stop rule, proof tier, run mode, autonomy window, budget cap, expected future run count, side-effect class, execution boundary, promotion authority, actual resource routes, and campaign state. Also create METRIC_STATE.json and METRIC_HISTORY.jsonl. Metric selection is part of the loop, not a permanent instruction chosen in advance.

Before every cycle, answer the metric-governor questions from current evidence: What outcome is this workflow ultimately trying to create for the user? What currently limits that outcome most: missing capability, weak quality, excess cost, unreliability, poor proof, bad structure, maintenance burden, or something else? Which observable signal best represents that bottleneck now? What valuable behavior could be destroyed by optimizing that signal harder? Has any current metric saturated, become easy to game, or stopped separating good from bad outputs? What did the latest success or failure reveal that the evaluator does not measure? Should the next operation ADD, REMOVE, RESTRUCTURE, VERIFY, ROUTE, STABILIZE, or HOLD complexity? What evidence would falsify the belief that this is the correct next bottleneck?

Record the answers in METRIC_STATE.json with a metric_epoch, immutable task purpose, hard constraints, current bottleneck, task-value dimensions, resource dimensions, grounding signals, metric hypothesis, known proxy risks, chosen operation, falsifier, and confidence. Freeze that metric state for the cycle before creating challengers. The model may propose a new metric, but it may not observe a candidate, invent a friendlier metric, and promote it retroactively.

The task purpose is stable unless the operator changes it. Metrics are mutable proxies for that purpose. A new metric epoch begins only between cycles, when evidence shows that the current proxy has saturated, become gameable, or decoupled from the actual outcome. The completed cycle remains judged under the metric state frozen for that cycle. Recalibrate the baseline and internal champion under the new metric before using it for promotion. A metric change that cannot be grounded in tests, external outcomes, user judgment, or another signal the optimizer does not control remains a hypothesis, not a promotion basis.

Perform preflight immediately. Read the baseline, its acceptance contract, real prior uses, failures, corrections, and available telemetry. Always surface the failure analysis and any prompt-level fix discoverable by reading. Decide whether a real comparison is reachable and whether the current evaluator can distinguish improvement. If controlled measurement is possible, static review is not a completed run. If measurement is genuinely unavailable, do not create projected logs, invented scorecards, dashboards, or scenario matrices. Keep the baseline hash, manifest, metric state, side-effect class, proof tier, failure map, readable candidate fix, and the exact evidence needed next. A reading-supported fix may be saved as a candidate, but it is not a measured promotion; classify it NEEDS_MEASUREMENT.

Choose the smallest proof tier that can honestly support the decision. TIER-MIN is for trivial, deterministic, low-risk targets where a full campaign would cost more than the likely gain; it still preserves the baseline hash, manifest, frozen metric state, direct before/after evidence, failure analysis, and decision. TIER-STANDARD adds the relevant resource map, working scenarios, raw run logs, scorecard, challengers, and compact dashboard. TIER-FULL is required for any measured promotion that is production-facing, subjective, high-risk, cross-session, resource-routing, multi-cycle, canonical, held-out/cold-check dependent, or noisy; it adds sealed evaluation records, judges, rollback notes, and resumable state. Right-size scaffolding, never truth.

Build the benchmark from real prior uses and failures where possible. Freeze an external task contract stating the intended outcome, required invariants, forbidden shortcuts, allowed side effects, proof requirements, and minimum acceptable quality. The target loop’s own Verify condition is evidence, not the sole oracle. Include representative successes, repaired failures, ambiguous inputs, and cases exercising any guard a challenger may remove. For research or web loops, freeze the source set or retrieval window before comparison. Simulated lookup tables may be controls, but not the sole promotion proof for a real agent workflow.

Freeze the working set, task contract, METRIC_STATE.json, scorecard, promotion rule, held-out set, and cold-check before challengers are written. Keep held-out and cold-check outside challenger-author context when tooling allows; exposed answer keys are development evidence, not sealed evidence. Held-out is for promotion validation, never tuning. Cold-check is a final one-shot test on the composed promotion candidate; never tune against it, and require a fresh cold set for any later repair cycle.

Map only resources that can materially change task value, reliability, or cost: model routes, tools, subagents, skills, scripts, browser paths, execution environments, context limits, quota pressure, marginal cost, speed, and reliability. Record the actual route and tooling used for every run. Undisclosed fallback or downgrade invalidates the comparison. If a challenger changes both workflow logic and resource routing, isolate the route change when practical or record the confound instead of claiming clean attribution.

Use a controller/worker structure for long campaigns so context does not collapse. The controller owns the manifest, metric state and history, champion status, deferred lanes, operator-stop state, and final promotion authority. Delegate lane runs, replays, trap sets, evaluator audits, held-out checks, and challenger families to fresh frontier-capable subagents or fresh runs when available, unless the operator or frozen resource policy explicitly specifies another route. Give each worker only the minimum packet needed: baseline hash, current champion or challenger, frozen scenarios, metric state, side-effect boundary, allowed tools, and success criteria. Worker outputs are untrusted evidence bundles: raw logs, artifact paths, measurements, failures, and lane verdicts. Workers may not promote, move metrics, erase attempts, alter the campaign objective, or mark the campaign complete. If fresh workers are unavailable, checkpoint to disk and resume from saved state in a fresh context.

Do not ask questions during the campaign. If a product, policy, risk, approval, account, secret, or external-judgment choice is needed, mark only that lane DECISION_GATED or DEFERRED with the exact unblock condition, choose the highest-value remaining runnable lane, and continue. Never offer "which lane should I run" or "want me to continue"; running open lanes is the default. If every lane is gated, platform-blocked, or safety-blocked, write STATE/CONTINUATION and wait for operator/retrigger without calling the campaign complete.

The scorecard must represent a vector, not a universal scalar. Hard constraints veto promotion first. Task-value dimensions determine whether the output is more useful for its actual purpose. Resource dimensions record tokens, dollars, wall time, tool calls, retries, repairs, maintenance, and human correction. Accepted-success cost remains a resource metric: total cost of every attempt, retry, repair, failed branch, context reload, and human correction divided by accepted successful outcomes; zero accepted successes means infinite cost. It is not automatically the primary objective for every task.

Before each cycle, declare how candidates will be compared. Use a lexicographic rule or Pareto rule rather than arbitrary weights unless weights have a defensible grounding. A candidate moves the frontier when it improves at least one task-value or resource dimension without unacceptable regression elsewhere, while satisfying all hard constraints. A lower-cost candidate that removes valuable complexity loses. A richer candidate may win at higher cost when the added value is observable and the tradeoff was allowed before testing. If task value and cost trade off without a predeclared decision rule, do not manufacture a scalar winner; stage the tradeoff for the appropriate external or human judgment.

Every scorecard value must trace to raw run evidence. Each run record must contain the actual command or tool trace, timestamp, model and route, provider usage or the observable proxy used, starting-state identifier, files read and changed, proof artifacts, result, and failure reason. Prefer harness telemetry over agent self-report. The scorecard must be reproducible from run_logs/; unproven runs are INCONCLUSIVE regardless of workspace polish. Estimates and lower bounds may support planning or NEEDS_MEASUREMENT, but never PASS, PROMOTE, READY_FOR_APPROVAL, or champion language.

Run the baseline through the real agent stack inside the declared boundary. Reset filesystem state, caches, fixtures, and controlled substitutes between arms, and use the same starting snapshot, task contract, metric state, scorecard, and resource policy. When order or stochasticity matters, counterbalance order and repeat enough to establish noise. For deterministic or bounded transforms, keep evidence lean: one reproducible harness and one results table may serve as the run log. The harness must execute real target behavior rather than return a prewritten answer, and at least one real agent execution must prove that the prompt survives the stack.

Record tokens when available, wall time, tool calls, subagents, files read and changed, retries, repairs, failed branches, human corrections, proof artifacts, task-value results, and pass/fail against the frozen contract. Agent or workflow failures remain in accepted-success cost. Runs invalidated by a benchmark, harness, or scorer defect remain in campaign-cost history but do not count against either arm; repair the instrument symmetrically, freeze it, and rerun every affected arm needed for a fair comparison.

Write failure_map.md before challengers. Identify not only where the baseline wastes cost or loses quality, but where the evaluator itself may be blind: vague steps, rereads, oversized context, missing capability, weak proof, bad routing, unnecessary subagents, late failure detection, unclear stopping, retry loops, drift, fragile assumptions, production-risk leakage, hidden rescue, user correction burden, saturated metrics, proxy gaming, or a grounding signal that no longer tracks the task purpose. Target the current bottleneck, not the easiest number to move.

When the loop parses or transforms external input, classify its archetype and run standard trap inputs before accepting “no material flaw” or NO PROMOTION. For text and CSV this includes a UTF-8 BOM, quoted embedded newlines, blank rows, quoted delimiters, and inconsistent columns; use analogous traps for numeric, time, path, encoding, stateful, and other archetypes. A no-flaw verdict without applicable traps is an unverified false negative.

Create up to three challengers per cycle. Each names the bottleneck, chosen operation, expected frontier movement, tradeoff, proof risk, resource change, immutable behavior, and falsifier. A challenger may add missing capability or evidence, remove waste, restructure the workflow, strengthen verification, change routing, stabilize variance, or deliberately preserve complexity that earns its cost. Longer, slower, or more complex is allowed when the added value is real and measured.

Run challengers on identical scenarios, execution boundaries, starting snapshots, task contracts, metric states, scorecards, and resource policies. Reject any challenger that changes task meaning, drops proof, skips evidence, uses hidden rescue, moves the yardstick, optimizes only prompt length, shifts burden into human correction, increases production risk, worsens severe-failure probability, conceals retries, undercounts attempts, or improves the proxy while degrading the grounded outcome. Preserve failed challengers and what they taught; a dead challenger is not a completed campaign.

Judge only observable artifacts: prompts, inputs, commands, files, tool outputs, screenshots, tests, telemetry, accepted results, downstream outcomes, and final artifacts. Ignore hidden reasoning and scratchpads. The metric judge applies the frozen scorecard. The task judge applies the frozen task contract and the loop’s Verify condition. The metric critic asks whether the chosen metrics still represent the task purpose and whether any proxy improved while the grounded outcome stalled or worsened. An independent critical reviewer checks intent, proof, boundaries, side effects, stopping behavior, evidence traceability, and gaming. For subjective or visual work, compare blinded artifacts side by side and use human or external feedback as a grounding signal when available.

The scorecard and metric state stay frozen within a cycle. If a scoring defect appears, log it. A clerical correction is allowed only when source evidence is unchanged, the correction is symmetric across all arms, and an independent reviewer shows that it cannot change promotion disposition. If it could change pass/fail, frontier status, quality floor, or promotion outcome, invalidate the affected comparison, freeze the corrected evaluator as a new metric epoch, and rerun it. Never promote from a moved yardstick.

After every cycle, perform metric reflection before choosing the next lever. Ask: Did the candidate improve the actual task outcome, or only the proxy? What useful behavior improved, disappeared, or became harder? Did the evaluator predict the result correctly? What important difference did it fail to notice? Has the proxy saturated or decoupled from the grounding signal? Is the current bottleneck still the same? If the metric remains valid, keep the epoch. If it failed, preserve the cycle verdict, open a new metric epoch, recalibrate baseline and champion, and continue under the revised evaluator. Never rewrite the completed cycle’s verdict to fit the new metric.

Only candidates that satisfy hard constraints and match or beat the baseline on required task value may face held-out. Advance when held-out confirms the claimed frontier movement and no serious new failure appears. Repeat noisy comparisons or mark them INCONCLUSIVE; never promote a lucky pass. Run cold-check once only after the final composed candidate earns promotion consideration.

Maintain champion discipline. The approved loop stays canonical until promotion authority permits replacement. The best measured candidate becomes the internal champion for later cycles, while the original baseline remains the rollback point and historical reference. If loop-file promotion is pre-authorized, promote a winner without another checkpoint: preserve a timestamped backup, verify old and new hashes, record candidate and evidence paths, and update the promotion log. That authority never expands to outside-world actions.

When an objective asks for multiple improvements, bank only independent measured frontier movements clearing the frozen bar. One root change measured on several axes counts as one improvement unless the operator explicitly defines separate banked wins; disclose causal dependence either way. After multiple banks, synthesize a combined candidate and test composition, held-out behavior, and one fresh cold-check before promotion.

Treat cycle verdicts and campaign state separately. PROMOTION, NO PROMOTION, INCONCLUSIVE, NEEDS_MEASUREMENT, BLOCKED, ECONOMICALLY_EXHAUSTED, DECISION_GATED, and DEFERRED apply to the current cycle or lane; they do not end the campaign. Continue from the internal champion with the next evidenced bottleneck. If platform, context, or budget interrupts execution, write STATE.md or CONTINUATION.md with the exact next cycle and mark ACTIVE / NEEDS_RESUME rather than complete.

Economic exhaustion prevents forced wins, not campaign stoppage. Record ECONOMICALLY_EXHAUSTED for a lane or metric epoch when no plausible operation there can move the frontier enough to justify its benchmark cost, repeated cycles expose no new testable bottleneck, all available metrics are below noise or lack grounding, held-out or cold-check regresses, or further work increases risk. Then continue another runnable lane or write ACTIVE / NEEDS_RESUME if none can run.

Update dashboard.html when the tier makes it useful. Show the immutable task purpose, active metric epoch, current bottleneck, chosen operation, baseline and internal champion, task-value results, accepted-success cost and other resource dimensions, grounding signal, proxy risks, working and held-out results, cold-check status, cycle verdict, campaign state, promotion status, and links to reproducible evidence. Multi-cycle campaigns maintain resumable state after every metric change, banked improvement, or killed challenger.

At each cycle end, report proof tier, side-effect class, execution boundary, actual routes, metric epoch, task purpose, bottleneck, operation, frozen comparison rule, task-value movement, resource movement, grounding result, failures fixed and open, challengers killed, improvements banked, internal and canonical champions, promotion disposition, campaign state, and exact next action; then execute the next action unless the operator has stopped you or the platform prevents execution. Never call the campaign complete yourself.

Verify / persistence

A cycle is valid only when the baseline is hash-locked, manifest and side-effect boundary exist, task purpose and metric state were frozen before challengers, baseline and challenger runs have raw evidence, all attempts count toward accepted-success cost, applicable traps ran, task value and resource cost were judged from observable artifacts, held-out and cold-check rules held, metric changes created new epochs rather than retroactive verdicts, and the promotion decision is reproducible.

A challenger wins only when the real agent stack moves the task-value/resource-cost frontier under the frozen metric state while preserving hard constraints. The canonical loop changes only under recorded authority. Campaign completion requires an operator stop; cycle failure, metric failure, gated decisions, deferred lanes, promotion, or economic exhaustion never silently become campaign completion.
